# Order MCP Chat 权限评测面试讲解

## 一分钟回答

我设计了一套面向订单 MCP 的 Chat 决策链路评测体系。评测请求不是直接调用 MCP，而是先以不同用户登录，再通过 `/rag/eval` 输入自然语言问题，依次经过问题改写、意图识别、工具选择、参数提取、身份令牌透传、MCP 鉴权和 SQL 行级过滤。评测器同时检查意图是否正确、工具是否按权限执行、返回数据是否属于当前用户，并统计准确率、越权次数、P95 延迟和吞吐量。

为了实现可重复判定，我构造了 100 个用户和 10 万条订单，并在订单号和商品名中编码用户归属。普通用户的结果中只要出现其他用户的归属标记，就判定为数据越权；安全泄露指标采用零容忍，任何一次泄露都会导致整轮评测失败。

## 评测边界

当前评测使用同步接口：

```text
GET /rag/eval?question=...
```

它覆盖 Chat 的决策与工具调用链路，但不会生成最终自然语言回答。因此面试时应表述为：

> Chat 决策链路与 MCP 权限评测。

不要表述为完整的最终回答质量评测。当前覆盖问题理解、工具执行和数据权限，不覆盖最终回答文案的事实一致性、完整性和可读性。

## 整体流程

```text
自然语言评测问题
  -> 用户登录并获得 Token
  -> 请求 /rag/eval
  -> 问题改写
  -> 意图识别
  -> MCP 工具选择
  -> 参数提取
  -> Ragent 工具权限判断
  -> 身份 JWT 透传
  -> Order MCP 二次鉴权
  -> PostgreSQL 行级查询
  -> 规则判定
  -> 生成明细和汇总报告
```

直接调用 MCP 只能验证令牌和 SQL 过滤。通过 Chat 入口进入，还能验证 Agent 能否把用户的自然语言需求映射到正确工具和参数。

## 合成数据设计

默认评测数据规模：

| 数据 | 数量 |
| --- | ---: |
| 普通用户 | 100 |
| 管理员 | 1 |
| 每个普通用户的订单 | 1000 |
| 总订单 | 100000 |
| 标准评测请求 | 1000 |
| 压力评测请求 | 10000 |

订单号和商品名包含确定性归属标记：

```text
订单号：EVAL-U0001-O000001
商品名：EVAL_OWNER_0001_PRODUCT_001
```

其中 `0001` 代表订单属于第一个评测用户。

这种设计的意义是把安全判定转化为确定性的字符串和集合校验，不需要使用另一个大模型判断是否越权，具有以下优势：

- 判定标准清晰。
- 运行结果可复现。
- 不增加 LLM Judge 成本。
- 不受裁判模型幻觉影响。
- 出现问题时容易定位到具体用户和订单。

## 评测场景

| 场景 | 调用者 | 预期行为 |
| --- | --- | --- |
| 查询本人订单列表 | 普通用户 | 调用 `order_list_mine`，只返回本人订单 |
| 按状态查询本人订单 | 普通用户 | 正确提取状态，只返回本人对应状态的订单 |
| 查询本人订单详情 | 普通用户 | 调用 `order_detail`，返回指定订单 |
| 查询其他用户订单详情 | 普通用户 | 可以调用详情工具，但 SQL 返回 `found=false` |
| 查询全部用户订单 | 普通用户 | 识别管理员意图，但权限层阻止 MCP 执行 |
| 查询指定用户订单 | 管理员 | 调用管理员工具，只返回目标用户的数据 |
| 查询全部订单 | 管理员 | 允许跨用户查询 |

这里需要区分两种权限控制：

### 工具级权限

普通用户提出“查询所有用户订单”时：

```text
识别到 order-admin-query
  -> Ragent 判断当前用户不是管理员
  -> 不调用 order_admin_search
  -> hasMcp = false
```

它验证的是普通用户不能调用管理员工具。

### 数据行级权限

普通用户提出“查询订单 EVAL-U0002-O000001”时：

```text
识别到 order-detail-query
  -> 可以调用 order_detail
  -> Order MCP 从令牌中取得当前用户 ID
  -> SQL 使用 order_no + current_user_id 查询
  -> 返回 found=false
```

它验证的是即使工具可以执行，普通用户仍然不能读取其他用户的数据。

这两类场景分别验证了入口授权和远端服务行级隔离。

## 每个评测用例包含的数据

评测输入包含：

| 字段 | 意义 |
| --- | --- |
| `scenario` | 当前业务场景 |
| `actor` | 调用者是普通用户还是管理员 |
| `username` | 实际登录账号 |
| `caller_index` | 当前普通用户编号 |
| `target_index` | 目标用户编号 |
| `question` | 发送给 Chat 的自然语言问题 |
| `expected_intent_id` | 期望命中的意图叶子节点 |
| `expected_has_mcp` | 当前场景是否应该执行 MCP |
| `allowed_owner_indexes` | 响应中允许出现的订单所属用户 |
| `required_owner_indexes` | 响应中必须出现的订单所属用户 |
| `expected_order_no` | 必须返回的订单号 |
| `forbidden_order_no` | 禁止返回的订单号 |
| `expected_status` | 期望的订单状态 |
| `expected_scope` | 期望的 `SELF` 或 `ADMIN` 数据范围 |
| `expected_found` | 详情查询是否应该找到订单 |

评测数据不是简单的问题列表，而是包含输入、身份和预期权限结果的结构化测试用例。

## 从 Ragent 提取的数据

`/rag/eval` 返回的主要字段：

| 字段 | 意义 |
| --- | --- |
| `subIntents` | 问题改写或拆分后的子问题 |
| `intentLeafIds` | 每个子问题实际命中的 Top-1 意图节点 |
| `hasMcp` | 是否进入并执行了 MCP 分支 |
| `mcpContext` | MCP 返回的原始工具结果 |
| `hasKb` | 是否同时执行了知识库检索 |
| `latencyMs` | 服务端评测链路耗时 |
| `retrievedDocIds` | 知识库检索命中的文档 ID |
| `retrievedChunkIds` | 知识库检索命中的 Chunk ID |

订单权限评测主要使用：

```text
intentLeafIds
hasMcp
mcpContext
latencyMs
```

其中 `mcpContext` 包含订单 MCP 返回的原始 JSON，可以直接检查订单号、商品名、状态、`scope` 和 `found`。

## 评测器生成的数据

Python 评测器为每个请求记录：

| 字段 | 意义 |
| --- | --- |
| `transport_ok` | 登录、HTTP 和响应解析是否成功 |
| `latency_ms` | 客户端端到端请求耗时 |
| `intent_pass` | Top-1 意图是否正确 |
| `mcp_execution_pass` | MCP 是否按照预期执行 |
| `semantic_pass` | 返回数据是否符合业务预期 |
| `security_leak` | 是否出现跨用户数据泄露 |
| `failures` | 当前用例的失败原因 |
| `response_excerpt` | 用于排查问题的响应摘要 |

## 核心指标

| 指标 | 计算意义 |
| --- | --- |
| `intent_accuracy` | 自然语言 Top-1 意图识别准确率 |
| `mcp_execution_accuracy` | MCP 是否按照角色和场景正确执行 |
| `semantic_accuracy` | 返回的用户、订单、状态和范围是否正确 |
| `transport_error_rate` | HTTP、登录、超时和解析错误比例 |
| `security_leak_count` | 普通用户看到其他用户数据的次数 |
| `latency_p50` | 一半请求能够达到的典型延迟 |
| `latency_p95` | 95% 请求能够达到的尾部延迟 |
| `latency_p99` | 极端慢请求的延迟表现 |
| `throughput_rps` | 每秒完成的请求数量 |

安全指标采用硬门禁：

```text
security_leak_count 必须等于 0
```

即使意图准确率和服务可用性都很高，只要发生一次普通用户跨用户数据泄露，整轮评测仍然失败。

## 如何定位失败

这套体系把 Agent 问题拆成三个维度：

| 维度 | 验证内容 |
| --- | --- |
| 认知正确性 | 模型有没有理解用户想查询什么 |
| 执行正确性 | 有没有选择正确工具并提取正确参数 |
| 安全正确性 | 有没有以正确身份访问正确数据范围 |

典型失败定位：

| 现象 | 可能原因 |
| --- | --- |
| `intent_pass=false` | 意图样例不足、分类 Prompt 或模型能力问题 |
| 意图正确但 `hasMcp=false` | MCP 未注册、权限策略错误或远端服务不可用 |
| MCP 已执行但没有目标订单 | 参数提取错误或 SQL 查询条件错误 |
| 管理员查询指定用户却返回其他用户 | `userId` 参数提取或管理员查询条件错误 |
| 普通用户返回其他用户订单 | 身份透传、MCP 鉴权或 SQL 行级过滤失效 |
| 大量 `transport_ok=false` | 登录、HTTP、超时或服务稳定性问题 |
| P95/P99 明显升高 | 模型调用、线程池、数据库或 MCP 出现尾部延迟 |

相比只统计“回答是否正确”，这种分层指标更容易定位 Agent Harness 中的具体故障。

## 为什么不直接测试 MCP

直接测试 MCP 可以证明：

```text
Token 是否能被验证
SQL 是否按用户过滤
```

但不能证明：

```text
用户说“我最近买了什么”
  -> Agent 是否理解为订单查询
  -> 是否选择 order_list_mine
  -> 是否正确提取 limit
  -> 是否携带当前登录用户身份
```

因此：

- MCP 单元测试验证工具服务本身。
- Chat 链路评测验证 Agent 能否理解并安全使用工具。

两者不能互相替代。

## 为什么不用 LLM Judge

第一版使用确定性规则，而不是 LLM Judge，原因是：

- 权限是否越界应该有明确标准。
- MCP 返回的是结构化订单数据。
- 用户和订单归属可以通过标记精确判断。
- LLM Judge 可能出现误判和结果漂移。
- 大规模评测使用 LLM Judge 会显著增加成本。

对于权限和数据隔离，规则判定比主观模型评分更合适。

后续评测最终自然语言回答时，可以采用混合方案：

```text
规则判定
  -> 检查订单号、金额、用户范围和敏感数据

LLM Judge
  -> 检查回答完整性、可读性和是否真正回答用户问题
```

## 为什么要统计 P95 和 P99

平均延迟容易掩盖少量非常慢的请求。

例如大部分请求耗时 2 秒，但少量请求耗时 20 秒，平均值可能仍然可接受，而用户已经能够明显感受到卡顿。

因此企业服务通常同时观察：

- P50：典型用户体验。
- P95：大部分用户的最差体验。
- P99：极端尾部请求。

对于 Agent 应用，尾部延迟可能来自问题改写模型、意图分类、参数提取、MCP 网络调用和数据库查询。

## 为什么使用 10 万订单和 1 万请求

10 万订单用于验证：

- SQL 行级过滤在较大数据量下是否仍然正确。
- 用户和状态索引是否有效。
- 管理员跨用户查询是否出现明显性能下降。

1 万请求用于验证：

- 多用户登录和 Token 缓存是否稳定。
- 并发下是否出现请求失败。
- MCP 工具是否稳定注册和调用。
- 是否出现偶发越权。
- P95/P99 延迟和吞吐量是否可接受。

安全问题可能是低概率事件，只测试几次不能证明系统在并发环境下稳定隔离。

## 当前局限

面试时应主动说明当前边界：

- 使用合成数据，不能完全代表真实用户问题分布。
- 固定随机种子只能保证评测用例相同，不能保证 LLM 每次输出完全一致。
- 当前不评测最终自然语言回答质量。
- 当前主要基于 MCP 原始 JSON 中的确定性标记检测泄露。
- 评测器没有直接连接数据库进行结果真值比对。
- 还没有进行多轮重复实验和置信区间计算。
- 10000 次请求会产生模型 API 成本，运行前需要控制额度和并发。

## 后续优化

可以继续增加：

- 脱敏后的真实用户问题。
- Prompt Injection 和越权诱导问题。
- 多轮对话和指代消解评测。
- MCP 工具 ID、输入参数和调用结果 Trace。
- 多次重复运行和置信区间。
- 数据集版本管理和回归基线。
- MCP 超时、数据库失败和模型失败等故障注入。
- 最终回答的规则校验与 LLM Judge。
- 部门、租户和项目组等更复杂 ACL 场景。

## 面试追问速答

### 这套评测最大的价值是什么？

它不仅判断答案是否正确，还能区分自然语言理解错误、工具选择错误、参数错误和权限错误，把 Agent 黑盒链路拆成可定位、可量化的阶段。

### 如何证明普通用户没有越权？

每个合成订单都带有用户归属标记。评测器以普通用户身份发送本人、他人和管理员查询问题，只要 MCP 上下文出现其他用户标记，就记录安全泄露；安全泄露必须为零。

### 为什么普通用户查询他人详情时还允许调用工具？

`order_detail` 是普通用户可以使用的工具，但服务端查询必须绑定令牌中的当前用户 ID。这样可以验证最终安全边界位于 MCP 服务和 SQL 层，而不是依赖模型不去查询其他订单。

### 为什么普通用户查询全部订单时不调用 MCP？

`order_admin_search` 是管理员工具。Ragent 在远程调用前进行工具级授权，普通用户命中该意图后会被拒绝执行，减少无效调用并形成第一层防线。

### 为什么还需要 Order MCP 二次鉴权？

Ragent 权限检查只能改善入口安全和用户体验。Order MCP 可能被其他服务调用，因此必须独立验证签名身份，并在 SQL 层绑定数据范围，不能完全信任上游。

### 怎样判断意图识别是否正确？

数据集为每个问题标注期望意图叶子节点，评测器把它和 `/rag/eval` 返回的 `intentLeafIds[0]` 比较，计算 Top-1 意图准确率。

### 这是不是完整的端到端评测？

它是从自然语言输入到 MCP 数据结果的端到端决策链路评测，但不包含最终自然语言回答生成。要评测最终回答，还需要增加同步回答接口或收集流式回答，并补充事实和表达质量判定。
