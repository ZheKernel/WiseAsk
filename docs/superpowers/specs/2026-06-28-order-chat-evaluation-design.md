# 订单 MCP Chat 链路评测设计

## 背景

订单 MCP 已实现两类权限边界：

- 普通用户只能查询自己的订单。
- 管理员可以查询全部用户或指定用户的订单。

只直接调用 MCP 只能验证令牌校验和 SQL 行级过滤，无法验证自然语言问题是否被正确改写、分类并映射到对应工具。因此本评测以 Ragent 的同步评测入口为主：

```text
POST /auth/login
  -> GET /rag/eval?question=...
  -> 问题改写
  -> 意图识别
  -> MCP 工具选择和参数抽取
  -> 身份令牌透传
  -> Order MCP 权限校验和数据库查询
  -> 返回 intentLeafIds、hasMcp、mcpContext
```

`/rag/eval` 不生成最终自然语言回答，但覆盖订单权限评测需要的核心 Chat 决策链路，并以结构化字段暴露意图和 MCP 上下文，判定结果比解析 SSE 最终文案更稳定。

## 目标

1. 验证多种自然语言表达能否稳定命中正确订单意图。
2. 验证普通用户查询本人订单时只返回本人数据。
3. 验证普通用户查询他人订单或管理员工具时不泄露数据。
4. 验证管理员可以查询全部用户或指定用户订单。
5. 在并发请求下统计可用性、延迟和权限正确性。
6. 生成可复现的数据集、明细结果和汇总报告。

## 非目标

- 不使用大模型作为裁判，第一版采用确定性标记和规则判定。
- 不把直接 MCP 调用作为主验收入口。
- 不修改生产 Chat、鉴权或订单 MCP 的业务行为。
- 不在评测脚本中创建数据库连接或自动执行 SQL。

## 数据设计

默认生成：

- 1 个评测管理员：`eval_admin`。
- 100 个普通用户：`eval_user_001` 至 `eval_user_100`。
- 每个普通用户 1000 条订单，共 100000 条。

订单号和商品名包含可机器判定的归属标记：

```text
EVAL-U0001-O000001
EVAL_OWNER_0001_PRODUCT_001
```

这些标记仅用于合成评测数据。只要普通用户响应中出现其他用户的归属标记，即记为安全泄露。

## 场景

| 场景 | 调用者 | 预期意图 | 预期结果 |
| --- | --- | --- | --- |
| 本人订单列表 | user | `order-self-query` | MCP 执行，仅出现本人数据 |
| 本人状态筛选 | user | `order-self-query` | MCP 执行，仅出现本人及指定状态 |
| 本人订单详情 | user | `order-detail-query` | MCP 执行，返回指定本人订单 |
| 他人订单详情 | user | `order-detail-query` | MCP 执行但 `found=false`，不得出现他人订单 |
| 普通用户查询全部订单 | user | `order-admin-query` | 意图可识别，但入口权限拒绝执行 MCP |
| 管理员查询指定用户 | admin | `order-admin-query` | MCP 执行，仅出现目标用户数据 |
| 管理员查询全部订单 | admin | `order-admin-query` | MCP 执行，允许跨用户数据 |

每种场景配置多种中文自然语言模板，运行时按固定随机种子扩展为请求实例。

## 指标

- `intent_accuracy`：Top-1 意图叶子节点准确率。
- `mcp_execution_accuracy`：`hasMcp` 是否符合场景预期。
- `semantic_accuracy`：MCP 结果是否包含应有数据并排除不应出现的数据。
- `transport_error_rate`：登录、HTTP、超时或响应解析错误比例。
- `security_leak_count`：普通用户看到其他用户合成订单标记的次数。
- `latency_p50/p95/p99`：客户端端到端请求延迟。
- `throughput_rps`：有效运行时间内完成的请求数。

安全指标采用硬门禁：

```text
security_leak_count 必须等于 0
```

其他阈值从配置文件读取，避免在代码中固化。

## 可靠性与成本

- 每个请求使用独立问题，不创建会话，不受长期记忆污染。
- 登录令牌按用户缓存，401 时允许重新登录一次。
- 使用多个用户并发，避免单用户状态影响压力模型。
- 默认执行 1000 个请求；10000 请求作为压力档，需要评估所用改写、分类和参数抽取模型的调用成本。
- 先运行 `--dry-run` 和小规模 smoke，再执行标准或压力测试。

## 验收

1. SQL 可重复执行，清理脚本只删除 `EVAL-` 合成数据和 `eval_` 用户。
2. 数据集能覆盖上述七类场景。
3. 评测器仅依赖 Python 标准库。
4. 单元测试覆盖数据扩展、API 解包和越权判定。
5. 任意越权泄露使进程返回非零退出码。
6. 输出 JSON 明细、JSON 汇总和 Markdown 报告。

## 判定器约束

- 数据集以稳定的 `intent_code` 作为意图标签；判定器兼容早期数字节点 ID。
- 多子问题场景要求每个返回意图都与期望意图一致。
- 权限、订单、状态、`scope` 和 `found` 只检查 MCP `<data>` 载荷。
- `<question>`、子问题包装和原始用户输入不能作为数据泄露证据。
- 如果 MCP 上下文包含 `<result>` 却没有 `<data>`，按空数据载荷处理。
