# 高风险区域和质量门禁

修改 RAG、Prompt、记忆、路由、数据、安全或管理端流程前，先读取本文件。

## Prompt 上下文

目标顺序：

```text
system 稳定上下文
半稳定上下文：长期记忆和会话摘要
近期原始历史
临时上下文：检索结果、MCP 结果、子问题和当前问题
```

规则：

- 不要把长期记忆或会话摘要混进动态证据里。
- 不要把用户记忆注入 system prompt。
- 检索证据和当前问题应保持在 Prompt 后段。
- 保留原始历史消息的角色。
- 模板变化需要测试消息顺序和 section 边界。

## 长期记忆

规则：

- 记忆抽取是机会性能力，不能阻塞聊天成功。
- 不要存储密钥、凭据、敏感个人属性或一次性临时事实。
- 召回必须有上限、可排序，并且格式紧凑。
- 行为变化时，要覆盖无效 JSON、禁用配置、upsert/merge、排序和 Prompt 位置。

## RAG Pipeline

典型流程：

```text
加载记忆
改写或拆分问题
召回长期记忆
识别意图
检索知识或调用 MCP
组装 Prompt 上下文
路由模型并流式回答
持久化消息
异步抽取记忆
```

规则：

- 保留降级行为和用户可见错误语义。
- 不要让可选的记忆或可观测逻辑阻塞流式输出。
- 小心取消请求、首 token 检测和持久化顺序。
- 改编排时要补阶段顺序测试。

## 检索和 MCP

规则：

- 新检索通道应遵循现有通道接口。
- 后处理要显式：去重、过滤、rerank、截断。
- Prompt 渲染中要区分 MCP 工具结果和知识库证据。
- 相关时测试空结果、重复结果、低置信度、畸形工具输出、超时和部分失败。

## 模型路由

规则：

- 健康状态、熔断、半开探测、优先级和降级顺序必须确定。
- 测试连续失败、恢复、禁用模型和降级顺序。
- 单个供应商失败不能污染其他供应商。

## 数据库

规则：

- Schema 变化应同步到 `resources/database/schema_pg.sql`，必要时增加版本化升级脚本。
- 尽量让迁移脚本可重复执行。
- 没有明确迁移和回滚说明时，不要删除或重写用户数据。
- Entity、Mapper、schema、初始化数据和测试要保持一致。

## 安全

规则：

- 不要提交真实 API Key 或凭据。
- 供应商 Key 和基础设施密钥优先使用环境变量。
- 检查用户 ID、租户、角色和资源归属边界。
- 不要把内部 trace、Prompt、私有检索内容或堆栈信息暴露给不可信客户端。

## 前端管理流程

验证：

- 加载、空状态、错误状态和成功状态。
- 表单校验和 API 失败行为。
- 桌面端和移动端布局。
- 权限敏感的导航。
- 没有不稳定布局跳动或文字重叠。
