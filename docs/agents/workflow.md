# 标准 Vibecoding 流程

实现非平凡改动时读取本文件。

## 开工门禁

先运行：

```powershell
git status --short --branch
```

判断工作区状态：

- 干净：创建分支并继续。
- 有无关用户改动：不要碰它们。
- 必须编辑的文件已有改动：先检查，再基于现有改动继续工作。

优先使用任务分支：

```powershell
git switch -c codex/<short-topic>
```

## 需求说明

非平凡工作需要创建或更新：

```text
docs/superpowers/specs/YYYY-MM-DD-<主题>-design.md
```

包含：

- 目标
- 非目标
- 当前行为
- 目标行为
- 设计方案
- 数据、API、Prompt 或 UI 变化
- 错误处理和降级行为
- 风险
- 验收标准
- 测试计划

高风险改动还需要迁移说明、回滚方案、必要时的功能开关或灰度说明，以及可观测
信号。

## 架构评审

实现前，先让高级模型或人工只评审架构。要求输出取舍、影响文件、测试和风险。
这个阶段不要让它写生产代码。

## 实施计划

创建或遵循：

```text
docs/superpowers/plans/YYYY-MM-DD-<topic>.md
```

每个任务都应写清楚文件、精确行为、步骤、预期测试、验证命令、代码差异检查范围和
建议提交信息。

## 执行

一次只实现一个任务。每个任务完成后运行：

```powershell
git diff --stat
git diff
```

先运行最小相关测试；如果影响范围更大，再扩展到相关回归测试。

## 模型路由

按风险选模型，不按价格选模型。

架构、SQL、鉴权、Prompt 组装、记忆、检索、模型路由、流式输出、不明确的失败
和最终评审，使用顶级模型或人工。

低价模型可以处理机械修改、行为明确的测试脚手架、文档清理、重复 UI 接线和受限
重命名。

## 提交策略

只提交连贯且已验证的检查点。

提交前：

```powershell
git diff --stat
git diff
git status --short
```

提交信息格式：

```text
<范围>: <行为变化>
```

示例：

- `prompt: 将记忆放在动态证据之前`
- `memory: 添加召回排序回归测试`
- `frontend: 校验知识库表单输入`
