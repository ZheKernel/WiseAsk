# 电商 Agent Harness 二次开发面试指南

## 1. 项目定位

本项目基于开源 Ragent 进行二次开发，原有能力包括文档入库、知识检索、意图识别、MCP 工具调用、模型路由和 SSE 流式问答。

二次开发重点解决三个企业 Agent 经常遇到的问题：

1. 用户进入新会话后，如何继续记住长期偏好、项目和约束。
2. 长期记忆、会话摘要、历史消息、知识库和工具结果如何有序进入 Prompt。
3. 多用户共享 Agent 时，如何保证每个用户只能检索和调用自己有权限的数据与工具。

为了证明权限不是只停留在设计层，还增加了电商订单 MCP 和 Chat 决策链路评测工程，形成“功能设计、业务实现、自动评测”的闭环。

## 2. 简历两句话

- 设计结构化长期记忆与四层 Prompt 上下文编排机制，基于 LLM 异步抽取 `PREFERENCE/PROJECT/CONSTRAINT/FACT` 记忆并按用户维度合并召回，结合会话摘要检查点和稳定前缀编排提升 KV Cache 复用率，兼顾跨会话上下文连续性与 Token 成本。
- 面向多用户电商订单场景构建 MCP 权限评测体系，基于短期 JWT 身份透传与 SQL 行级过滤实现管理员和普通用户隔离，生成 100 用户、10 万订单的合成数据集并支持扩展至 1 万次 Chat 决策链路请求，覆盖意图识别、工具调用、P95/P99 延迟与越权检测；首轮 Chat 链路实测中 MCP 跨用户数据泄露 0 次。

这里的“支持 1 万请求评测”表示评测器和数据集能够按该规模执行，不应表述为已经取得某个 1 万请求压测结果。没有正式报告时，不要虚构吞吐量或准确率。

## 3. 面试开场回答

> 我主要从 Agent Harness 角度对 Ragent 做了三部分增强。第一部分是长期记忆，让系统在跨会话场景下能够按用户保存偏好、项目背景和长期约束，并在下一次提问时只召回少量相关记忆。第二部分是 Prompt 上下文编排，把系统规则、长期记忆、会话摘要、近期历史和本轮 RAG/MCP 证据按稳定性分层，既保持优先级边界，也有利于稳定前缀的 KV Cache 复用。第三部分是权限与评测，我为知识库增加全局和个人作用域，又通过短期 JWT 把登录身份透传到独立订单 MCP，在服务端和 SQL 层做二次鉴权；最后构造 100 用户、10 万订单的电商数据集，从 Chat 入口验证自然语言意图、工具调用和零越权。

## 4. 模拟电商业务场景

系统模拟一个面向消费者、客服和平台管理员的电商智能助手。

### 4.1 普通用户

普通用户可以：

- 上传自己的发票、售后记录和使用说明，形成个人知识库。
- 查询自己的历史订单、订单状态和商品信息。
- 让系统记住长期偏好，例如预算、品牌偏好和常用技术栈。
- 在新会话中继续使用之前积累的项目和偏好信息。

普通用户不能：

- 检索其他用户上传的个人资料。
- 查看其他用户订单。
- 调用管理员订单查询工具。
- 通过 Prompt Injection 伪造 `userId` 或管理员角色。

### 4.2 管理员

管理员可以：

- 上传平台售后政策、物流规则和公共商品手册，形成全局知识库。
- 在管理端查看和管理全部知识库。
- 通过管理员订单工具查询全部用户或指定用户订单。

管理员普通聊天默认仍然只检索：

```text
GLOBAL + 管理员自己的 PERSONAL
```

它不会自动把所有用户个人资料放进 Prompt。跨用户知识检索需要单独的审计入口，不能因为角色是管理员就默认扩大普通 Chat 的上下文范围。

### 4.3 组合问题

例如用户提出：

```text
我上次购买的机械键盘是否还在平台保修期内？
```

理想链路包括：

```text
问题改写和拆分
  -> Order MCP 查询当前用户最近的机械键盘订单
  -> 全局知识库检索平台保修政策
  -> 用户长期记忆补充品牌或设备偏好
  -> 只使用已授权证据组装 Prompt
  -> 模型生成回答
```

这个场景体现了 Memory、RAG、MCP 和 Permission 不是独立功能，而是共同参与一次 Agent 执行。

## 5. Harness Engineering 维度

Harness Engineering 的目标不是继续训练模型，而是在模型外部控制：

```text
给模型什么上下文
允许模型调用什么工具
工具以谁的身份执行
失败时如何降级
结果如何评测和审计
```

本项目可以归纳为六个维度：

| 维度 | 本项目实现 |
| --- | --- |
| Model | 多模型路由、健康状态、熔断和降级 |
| Prompt/Context | 四层 Prompt 编排和稳定前缀 |
| Memory | 会话窗口、摘要检查点和结构化长期记忆 |
| Tool/MCP | 意图驱动工具选择、参数提取和订单 MCP |
| Permission | 知识库作用域、工具授权、JWT 和 SQL 行级过滤 |
| Runtime/Evaluation | SSE、异步任务、Trace、业务数据集和权限评测 |

## 6. 功能一：结构化长期记忆

### 6.1 原有问题

原项目的会话消息和会话摘要主要解决单个会话内的上下文增长。

用户开启新会话后，以下信息仍然会丢失：

- 长期技术偏好。
- 持续维护的项目。
- 用户明确提出的长期约束。
- 后续问题会重复使用的稳定事实。

如果每次都携带全部历史消息，又会带来：

- Token 成本不断增长。
- 无关历史干扰当前问题。
- Prompt 前缀频繁变化。
- 模型注意力被大量旧信息稀释。

### 6.2 记忆类型

长期记忆分为：

| 类型 | 含义 | 电商示例 |
| --- | --- | --- |
| `PREFERENCE` | 长期偏好 | 偏好静音机械键盘 |
| `PROJECT` | 持续任务或项目 | 正在组装家庭工作站 |
| `CONSTRAINT` | 长期约束 | 推荐商品预算不超过 1000 元 |
| `FACT` | 用户明确提供的稳定事实 | 当前使用 Windows 系统 |

### 6.3 抽取链路

```text
助手回答完成
  -> 持久化 assistant 消息
  -> LongTermMemoryService.extractAsync
  -> 独立线程池读取当前用户近期对话
  -> LLM 按固定 JSON Schema 输出候选记忆
  -> 校验类型、Key、长度、置信度和重要度
  -> 写入或合并 t_user_long_term_memory
```

数据库记录：

- `user_id`
- `memory_type`
- `memory_key`
- `content`
- `confidence`
- `importance`
- 来源会话和消息
- 访问次数
- 最近访问时间
- 状态

唯一约束以：

```text
user_id + memory_type + memory_key + deleted
```

避免同一用户的相同记忆不断重复写入。

### 6.4 合并策略

当新旧记忆 Key 相同时：

- 新内容更具体时更新内容。
- 置信度和重要度保留较高值。
- 保留访问统计和来源信息。
- 不直接新增一条重复记忆。

第一版使用确定性合并，而不是继续调用 LLM 决定是否合并，原因是更便宜、可解释且容易写测试。

### 6.5 召回链路

```text
用户发起新问题
  -> 完成问题改写
  -> 只查询当前 user_id 的 ACTIVE 记忆
  -> 计算词法相关性
  -> 叠加重要度、置信度和访问次数
  -> 排序并截取 Top N
  -> 格式化为长期记忆上下文
```

选择在问题改写后召回，是因为改写后的问题通常会补全口语、省略和指代，更适合作为检索键。

### 6.6 为什么第一版不使用向量检索

用户长期记忆数量较小时，关系库和确定性排序有以下优势：

- 不需要额外 Embedding 成本。
- 排序逻辑可解释。
- 用户隔离容易验证。
- 测试不依赖向量服务。
- 数据规模增大后仍可在前面增加向量候选召回。

因此当前选择是第一阶段的工程取舍，不是认为长期记忆永远不需要向量检索。

### 6.7 可靠性设计

长期记忆属于增强能力：

- 抽取使用独立线程池。
- 抽取发生在回答完成之后。
- 无效 JSON 和未知类型直接忽略。
- 抽取异常只记录日志。
- 召回异常降级为空记忆。

记忆失败不能阻塞用户得到回答。

## 7. 功能二：四层 Prompt 上下文编排

### 7.1 为什么要分层

不同上下文的稳定性和可信度不同：

- 系统规则最稳定且优先级最高。
- 长期记忆和摘要相对稳定，但来自用户数据。
- 近期历史会不断增长。
- RAG 和 MCP 证据只对当前请求有效。

如果全部拼成一个动态字符串，会造成：

- 上下文优先级不清晰。
- 用户记忆可能污染系统规则。
- 本轮证据和历史事实混在一起。
- Prompt 前缀频繁变化，降低 KV Cache 复用机会。

### 7.2 四层结构

| 层级 | 内容 | 特点 |
| --- | --- | --- |
| Stable | System Prompt、角色和安全规则 | 长期稳定 |
| Semi-stable | 长期记忆、会话摘要 | 跨多轮复用 |
| History | 摘要边界后的近期原始消息 | 保留角色顺序 |
| Ephemeral | KB/MCP 证据、子问题和当前问题 | 每次请求变化 |

最终顺序：

```text
System 稳定规则
  -> 当前用户长期记忆
  -> 当前会话摘要
  -> 近期原始消息
  -> 已授权的 KB/MCP 证据
  -> 当前问题
```

### 7.3 为什么长期记忆不放进 System Prompt

长期记忆来自用户对话，可信度低于系统规则。

放入 System Prompt 会：

- 错误提升用户数据的优先级。
- 增加记忆污染系统行为的风险。
- 让用户通过对话间接修改系统规则。

因此长期记忆作为 Semi-stable 用户上下文，而不是系统指令。

### 7.4 会话摘要检查点

摘要不是每轮都重写，而是在达到轮次或 Token 阈值后增量更新。

摘要更新后：

- 保存摘要覆盖到哪条消息。
- 保留检查点之后的近期原始消息。
- Prompt 使用摘要加近期窗口，而不是完整历史。

这样既控制上下文长度，又保留最近对话的原始角色和细节。

### 7.5 KV Cache 如何体现在设计中

KV Cache 通常由模型服务或推理框架提供，项目本身没有重新实现模型侧 KV Cache。

本项目做的是：

- 把系统规则放在稳定前缀。
- 让长期记忆和摘要保持相对稳定。
- 把变化最大的 RAG、MCP 证据和当前问题放在后段。
- 避免无意义地改变前缀格式和顺序。

这种稳定前缀编排有利于支持 Prefix/KV Cache 的模型服务复用已有前缀计算。

面试时应说“提升 KV Cache 复用机会”，不要说“自研了 KV Cache”。

## 8. 功能三：知识库权限控制

### 8.1 业务需求

电商平台存在两类知识：

- 管理员上传的公共售后政策、物流规则和商品手册。
- 用户上传的个人发票、订单凭证和设备资料。

系统必须确保其他用户的个人文档不会进入当前用户 Prompt。

### 8.2 数据模型

知识库增加：

```text
scope = GLOBAL / PERSONAL
owner_user_id
```

权限矩阵：

| 操作 | 普通用户 | 管理员 |
| --- | --- | --- |
| 创建知识库 | 只能创建自己的 `PERSONAL` | 可创建 `GLOBAL/PERSONAL` |
| 管理知识库 | 只能管理自己的个人库 | 管理端可管理全部 |
| 普通 Chat 检索 | `GLOBAL + 自己的 PERSONAL` | `GLOBAL + 自己的 PERSONAL` |

### 8.3 权限链路

```text
Sa-Token 当前登录用户
  -> RagResourcePermissionService
  -> 计算授权 Collection
  -> 写入 SearchContext.authorizedCollections
  -> 全局向量通道只遍历授权集合
  -> 意图定向通道过滤未授权 Collection
  -> 只有授权 Chunk 可以进入 Prompt
```

权限必须在数据进入模型上下文之前完成。不能把越权内容先交给模型，再要求模型不要回答。

### 8.4 为什么不能只在前端控制

只隐藏按钮无法阻止：

- 用户直接调用后端接口。
- 绕过知识库接口直接访问文档或 Chunk 接口。
- 意图节点指向未授权 Collection。
- 全局向量搜索扫描其他用户数据。

因此知识库、文档、Chunk、检索通道和 Prompt 输入均有服务端权限边界。

## 9. 功能四：受保护的订单 MCP

### 9.1 为什么模型参数不能代表用户身份

LLM 生成的工具参数属于不可信输入。

如果订单工具直接接受：

```json
{"userId":"其他用户ID"}
```

攻击者可以通过 Prompt Injection 诱导模型传入其他用户 ID。

用户身份必须来自服务端登录上下文，而不是模型参数。

### 9.2 身份透传

```text
Ragent 当前 LoginUser + Sa-Token
  -> 检查工具权限
  -> 使用 Ragent 私钥生成 private_key_jwt Client Assertion
  -> Auth Server 验证 Client JWKS、Sa-Token 会话和 t_user
  -> Auth Server 签发 RS256 Access Token
  -> Authorization: Bearer <access-token>
  -> Spring Security Resource Server + OrderMcpIdentityBridgeFilter
  -> 校验签名、issuer、audience、scope 及令牌生效/过期时间
  -> McpCallerIdentity 写入 TransportContext
```

JWT 包含：

- `sub`
- `username`
- `role`
- `scope`
- `client_id`
- `iss`
- `aud`
- `iat`
- `nbf`
- `exp`
- `jti`

### 9.3 双层鉴权

第一层位于 Ragent：

- 判断当前用户能否调用目标工具。
- 普通用户不能调用 `order_admin_search`。
- 无权限时不发起远程请求。

第二层位于 Order MCP：

- 通过 Auth Server JWKS 独立验证 JWT。
- 再次检查 Scope 和管理员角色。
- SQL 强制绑定令牌用户 ID。

即使绕过 Ragent 直接请求 MCP，也不能绕过最终权限边界。

### 9.4 三个订单工具

| 工具 | 服务端规则 |
| --- | --- |
| `order_list_mine` | 不接受用户 ID，始终按令牌 `sub` 查询 |
| `order_detail` | 普通用户按 `order_no + token.sub` 查询 |
| `order_admin_search` | 仅管理员可用，可指定用户和状态 |

订单服务只暴露固定、参数化、只读 SQL，并限制最大返回数量。

### 9.5 为什么使用短期 JWT

短期 JWT 提供：

- 身份完整性。
- 角色不可被模型篡改。
- 过期时间。
- 服务受众限制。
- 独立服务验证能力。

JWT 解决的是跨服务身份可信问题。Ragent 和 Order MCP 是两个独立进程，远端服务无法直接读取 Ragent 的 `UserContext`，因此需要一个可验证的身份载体。

不能直接传 `userId` 的原因是：

- `userId` 可能来自 LLM 工具参数，模型和用户都能影响它。
- 普通 JSON 参数没有签名，调用方可以随意修改角色和用户 ID。
- Order MCP 无法判断该身份是否真的由 Ragent 签发。

JWT 将 `sub`、`role`、`aud` 和过期时间一起签名。修改任何 Claim 都会导致验签失败，Order MCP 可以在不共享 Ragent 登录 Session 的情况下独立恢复调用者身份。

JWT 也有明确边界：

- JWT 默认只签名，不加密，敏感内容仍然需要 TLS 保护。
- JWT 只完成认证信息传递，不能代替工具授权和 SQL 行级过滤。
- `jti` 只是令牌唯一标识；没有服务端重放缓存时，不能单独防止令牌在有效期内被重放。
- 令牌泄露后的风险由短 TTL 限制，但不能完全消除。

### 9.6 为什么不直接透传用户登录 Token

用户登录 Ragent 使用的 Token 面向主应用，内部 MCP Token 面向具体 MCP 服务，两者职责不同。

直接把用户登录 Token 转发给所有 MCP 会带来：

- 原始登录凭证暴露给更多下游服务，扩大泄露半径。
- Token 的受众通常是 Ragent，而不是订单服务。
- 下游服务可能获得超出自身需要的用户信息和权限。
- 不同 MCP 无法获得独立的有效期、Scope 和 Audience。

本项目采用类似令牌交换的思路：

```text
用户登录 Token
  -> Ragent 验证并得到 LoginUser
  -> Ragent 根据目标 MCP 签发短期 audience-specific JWT
  -> Order MCP 只接受 aud=order-mcp 的内部令牌
```

这样下游拿到的是最小范围、短期有效的委托身份，而不是用户的原始登录凭证。

### 9.7 企业中的常见身份传递方式

企业通常不会让 Agent 自己维护一套孤立的长期共享密钥，而是接入统一身份基础设施：

```text
用户
  -> 企业 IdP 登录并获得 Access Token
  -> Agent Gateway 验证用户身份
  -> Policy Engine 判断可调用的 MCP 和工具
  -> STS / Token Exchange 签发目标服务专用短期 Token
  -> 通过 mTLS 调用 MCP
  -> MCP 校验客户端身份、用户身份、Audience 和 Scope
  -> 业务服务执行租户或用户级数据过滤
```

常见技术选择：

| 目标 | 企业常见方案 |
| --- | --- |
| 用户认证 | OAuth 2.0、OIDC、企业 SSO |
| 服务身份 | OAuth 2.0 Client Credentials、云 IAM、SPIFFE/SPIRE |
| 用户委托 | OAuth 2.0 Token Exchange、On-Behalf-Of |
| Token 签名 | RSA/ECDSA 非对称签名，通过 JWKS 发布公钥 |
| 通道认证 | mTLS、Service Mesh |
| 权限决策 | RBAC、ABAC、OPA、Casbin 或企业 IAM Policy |
| 密钥管理 | Vault、KMS、自动轮换 |
| 审计 | API Gateway、Trace、SIEM 和安全告警 |

当前项目使用共享 HMAC，适合本地开发和演示。生产环境更推荐非对称签名：身份平台持有私钥，MCP 只持有公钥，MCP 即使被攻破也不能反向签发管理员令牌。

### 9.8 MCP 如何防止其他系统随意接入

首先要明确：知道 MCP 地址不应该等于能够调用 MCP，但只增加 JWT 也不等于完成了客户端准入。

当前项目的防护链路是：

```text
Ragent 根据当前 LoginUser 检查目标工具权限
  -> 签发目标 MCP 专用的短期 JWT
  -> 访问 /mcp
  -> 缺少 Bearer Token 直接返回 401
  -> 校验 HS256 算法、签名和生效时间
  -> 校验 issuer 和 audience
  -> 将已验证身份写入 MCP TransportContext
  -> Order MCP 再检查 user/admin 角色
  -> SQL 按 token.sub 绑定数据范围
  -> 记录调用用户、角色、工具、结果数量和耗时
```

服务启动时使用 `system` 身份完成 MCP 初始化和 `tools/list`。该身份虽然能通过协议握手，但不是 `user/admin` 业务身份，执行订单查询时会被 `OrderQueryService` 拒绝。

当前实现可以阻止：

- 没有 Token 的客户端。
- 使用伪造签名、错误 Issuer、错误 Audience 或过期 Token 的客户端。
- 普通用户调用管理员工具。
- 通过工具参数伪造其他 `userId` 的查询。

当前实现仍有生产级缺口：

- 任何持有 HMAC 共享密钥的系统都能自行签发管理员 Token。
- 如果 `/mcp` 暴露到公网，仅靠应用层 JWT 会承受扫描和攻击流量。
- 当前 `jti` 没有接入重放检测存储。
- 当前没有基于客户端证书或工作负载身份的服务白名单。

企业通常增加以下防线：

1. MCP 部署在 VPC、Kubernetes `ClusterIP` 或内网域名中，通过安全组和 `NetworkPolicy` 限制来源。
2. API Gateway 或 Service Mesh 使用 mTLS 验证调用方工作负载身份。
3. OAuth2 客户端注册或工作负载身份控制“哪些系统可以申请 Token”。
4. JWT 使用独立 Audience 和最小 Scope，例如 `order:read:self`、`order:read:any`。
5. MCP 服务端继续执行工具级 RBAC/ABAC，不能只依赖网关。
6. 数据库层继续执行租户、用户行级过滤，必要时使用 PostgreSQL RLS。
7. 增加限流、配额、请求大小限制、Schema 校验和异常调用熔断。
8. 高风险操作使用短 TTL、`jti` 重放检测、审批和幂等键。
9. 将调用方、用户、工具、参数摘要和结果范围写入不可抵赖审计日志。

面试时可以总结为：

> JWT 解决用户身份在跨服务调用中的防篡改传递，mTLS 和网络策略解决调用方准入，工具授权解决能调用什么，SQL 或 RLS 解决最终能看到哪些数据。

## 10. 功能五：电商订单业务评测

### 10.1 评测体系的总体思路

这套评测不是先生成大量问题再看成功率，而是从风险和业务不变量反推测试集。

第一步，定义必须始终成立的业务不变量：

- 普通用户只能看到自己的订单。
- 普通用户不能执行管理员工具。
- 管理员可以查询全部订单或指定用户订单。
- 他人订单号即使被正确提取，也必须返回 `found=false`。
- 任何身份、权限或下游异常都必须失败关闭，不能返回未授权数据。

第二步，把每条不变量转换为正向、反向和越权用例：

| 风险 | 对应用例 |
| --- | --- |
| Agent 无法理解用户需求 | 本人列表、状态筛选、订单详情等自然语言变体 |
| 问题改写丢失身份语义 | “我的已付款订单”等带所有权表达的问题 |
| 工具选择错误 | 普通用户问题被错误路由到管理员工具 |
| 参数越权 | 普通用户输入其他用户订单号 |
| 角色伪造 | 普通用户要求“以管理员身份查询全部订单” |
| SQL 过滤失效 | 工具结果出现其他用户的订单归属标记 |
| 服务不稳定 | 并发请求下的超时、错误率和尾部延迟 |

第三步，为每个用例设计机器可判定的真值，而不是依赖人工阅读回答：

- 固定 `intent_code` 作为期望意图。
- 标注是否应该执行 MCP。
- 标注允许和禁止出现的订单归属。
- 标注期望订单号、状态、`scope` 和 `found`。
- 使用确定性的订单号、商品名和用户 ID 归属标记。

第四步，将结果拆成独立维度。意图失败、工具失败、业务语义失败、网络失败和安全泄露分别统计，避免只看一个总分而无法定位问题。

第五步，设置不同性质的门禁：

- 准确率和错误率使用阈值。
- P95/P99 用于发现尾部延迟。
- 安全泄露采用硬门禁，`security_leak_count` 必须为 0。

完整评测链路：

```text
固定随机种子生成结构化用例
  -> 使用不同普通用户和管理员账号登录
  -> 从 /rag/eval 输入自然语言问题
  -> 经过问题改写、意图识别、工具选择和 MCP 调用
  -> 采集接口结果与客户端耗时
  -> 确定性判定器分别判断意图、工具、语义和越权
  -> 汇总分场景准确率、错误率、P50/P95/P99 和吞吐量
  -> 阈值检查并输出 JSON/Markdown 报告
```

### 10.2 为什么从 Chat 入口测试

直接调用 MCP 只能验证：

```text
Token 是否有效
SQL 是否过滤用户
```

从 `/rag/eval` 进入还可以验证：

```text
自然语言理解
  -> 问题改写
  -> 意图识别
  -> 工具选择
  -> 参数提取
  -> 权限判断
  -> MCP 执行
  -> 数据范围
```

当前 `/rag/eval` 不生成最终自然语言回答，因此准确名称是“Chat 决策链路评测”，不是最终回答质量评测。

### 10.3 数据规模

| 数据 | 数量 |
| --- | ---: |
| 管理员 | 1 |
| 普通用户 | 100 |
| 每用户订单 | 1000 |
| 总订单 | 100000 |
| 标准评测请求 | 1000 |
| 压力评测能力 | 10000 请求 |

订单归属使用确定性标记：

```text
EVAL-U0001-O000001
EVAL_OWNER_0001_PRODUCT_001
```

### 10.4 评测场景

| 场景 | 主要验证 |
| --- | --- |
| 本人订单列表 | 自然语言能否命中本人订单工具 |
| 本人状态筛选 | 是否保留“我的”并正确提取状态 |
| 本人订单详情 | 订单号参数提取是否正确 |
| 他人订单详情 | 行级过滤是否返回 `found=false` |
| 普通用户管理员查询 | 工具级权限是否失败关闭 |
| 管理员指定用户查询 | 参数和目标用户范围是否正确 |
| 管理员全量查询 | 管理员跨用户权限是否生效 |

### 10.5 用例生成与执行方式

Python 评测器从模板文件加载场景，并使用固定随机种子和场景权重生成指定数量的用例。固定种子可以保证每次抽到的用户、订单和场景一致，方便回归对比。

每条用例包含：

- `case_id` 和场景类型。
- 登录账号和调用者角色。
- 自然语言问题。
- 期望 `intent_code`。
- 是否期望执行 MCP。
- 允许和必须出现的订单归属。
- 禁止出现的外部订单号。
- 期望状态、`scope` 和 `found`。

执行阶段：

```text
按用户名登录并缓存 Ragent Token
  -> ThreadPoolExecutor 按配置并发执行
  -> GET /rag/eval?question=...
  -> 401 时清理缓存并重新登录一次
  -> 保存服务返回字段和客户端端到端耗时
  -> Judge 生成 CaseResult
```

评测账号不是只换请求参数，而是真正使用不同账号登录，因此能够覆盖登录身份到 MCP 内部 JWT 的完整透传链路。

### 10.6 评测提取的数据

接口返回：

- `subIntents`
- `intentLeafIds`
- `hasMcp`
- `mcpContext`
- `hasKb`
- `latencyMs`

评测器生成：

- `intent_pass`
- `mcp_execution_pass`
- `semantic_pass`
- `security_leak`
- `transport_ok`
- 客户端端到端耗时
- 失败原因和响应摘要

### 10.7 判定策略

数据集使用稳定的 `intent_code`：

```text
order-self-query
order-detail-query
order-admin-query
```

评测器兼容早期数字节点 ID。

权限判定只检查 MCP `<data>` 载荷：

```xml
<result>
  <question>用户原始或拆分后的问题</question>
  <data>真正的工具结果</data>
</result>
```

`<question>` 中出现其他订单号不代表泄露。只有 `<data>` 返回其他用户订单、商品归属标记或用户 ID，才记为安全泄露。

判定器分别执行：

- 意图判定：所有拆分后的叶子意图都必须与期望 `intent_code` 一致。
- 工具判定：`hasMcp` 必须与该场景的预期执行状态一致。
- 语义判定：检查订单归属、订单号、状态、`scope` 和 `found`。
- 安全判定：普通用户的 `<data>` 中不能出现其他用户归属。
- 传输判定：登录、HTTP、超时和 JSON 解析异常单独记录。

安全判定使用确定性规则而不是 LLM Judge，因为订单归属和权限边界有明确真值。这样可以降低模型评判成本，并避免 Judge 幻觉把问题文本误判为数据泄露。后续评测最终自然语言回答时，再引入规则加 LLM Judge 的混合方案。

### 10.8 核心指标

| 指标 | 意义 |
| --- | --- |
| `intent_accuracy` | 自然语言 Top-1 意图准确率 |
| `mcp_execution_accuracy` | 工具是否按角色正确执行 |
| `semantic_accuracy` | 返回订单、用户和状态是否正确 |
| `transport_error_rate` | 登录、HTTP、超时和解析失败比例 |
| `security_leak_count` | 普通用户看到其他用户数据的次数 |
| `latency_p50/p95/p99` | 典型和尾部延迟 |
| `throughput_rps` | 每秒完成请求数 |

安全指标采用零容忍：

```text
security_leak_count 必须等于 0
```

### 10.9 分层执行策略

评测按从低成本到高成本逐层执行：

| 层级 | 验证内容 | 目的 |
| --- | --- | --- |
| 单元测试 | JWT Codec、鉴权 Filter、订单权限、数据集和 Judge | 先验证确定性组件 |
| 手工集成 | 登录、工具注册、本人/他人/管理员查询 | 排除配置和服务问题 |
| Dry Run | 只生成用例，不调用模型 | 检查场景分布和真值 |
| 20 请求 Smoke | 完整 Chat 决策链路 | 快速暴露编排和判定问题 |
| 1000 请求标准评测 | 准确率、错误率和尾部延迟 | 形成稳定基线 |
| 10000 请求压力评测 | 吞吐、稳定性和资源瓶颈 | 验证容量，不作为首轮运行 |

当前已经有明确结果的是评测器 15 个单元测试和首轮 20 请求 Smoke 复核。1000 和 10000 请求是评测工程支持的标准与压力规模，只有实际运行并保存报告后，才能写成简历中的实测结果。

### 10.10 首轮 Smoke 测试的工程价值

首轮 20 请求 smoke 报告曾暴露两类问题：

1. 评测器使用数字节点 ID 与接口返回的 `intent_code` 比较，造成全量意图误报。
2. 多子问题包装中的 `<question>` 出现目标订单号，被错误当成数据泄露。

修复后：

- 新数据集直接使用 `intent_code`。
- 判定器兼容旧数字 ID。
- 只扫描 `<data>` 载荷。
- 多子问题要求所有意图与期望一致。
- 真正的外部用户订单进入 `<data>` 时仍触发硬失败。

对首轮样本复核后，20 条中有 18 条意图正确，原先报告的 1 次越权为误报；剩余两条真实失败来自问题改写将“我的”删除，导致本人状态查询被识别为管理员查询。权限层最终拒绝了管理员工具，因此没有发生数据泄露。

这个过程体现了评测体系本身也需要回归测试，不能把评测器输出无条件当成真值。

## 11. 完整业务链路

```text
用户通过 Sa-Token 登录
  -> 加载会话摘要和近期历史
  -> 问题改写与拆分
  -> 按 user_id 召回长期记忆
  -> 计算 GLOBAL + 当前用户 PERSONAL Collection
  -> 意图识别
  -> 授权知识库检索 / 鉴权 MCP 调用
  -> 只使用已授权证据组装四层 Prompt
  -> 模型路由与流式回答
  -> 消息持久化
  -> 异步抽取长期记忆
  -> Trace 和评测报告记录结果
```

## 12. 共同设计原则

### 12.1 先权限，后 Prompt

知识片段和工具结果在进入 Prompt 前必须完成授权。

### 12.2 模型输入不等于可信身份

LLM 可以决定业务参数，但不能决定用户身份和角色。

### 12.3 增强能力可降级，安全能力失败关闭

- 记忆失败返回空上下文。
- 权限判断异常时不返回私有数据。

### 12.4 主链路与后台任务隔离

流式回答优先，长期记忆抽取异步执行。

### 12.5 评测必须能够定位问题

不能只统计“回答对不对”，还要分别观察：

- 意图是否正确。
- MCP 是否执行。
- 参数和语义是否正确。
- 是否越权。
- 是否出现传输和性能问题。

## 13. 关键技术取舍

| 问题 | 当前选择 | 原因 |
| --- | --- | --- |
| 长期记忆检索 | 关系库和确定性排序 | 便宜、可解释、容易测试 |
| 记忆抽取 | 回答完成后异步 LLM | 不阻塞主链路 |
| Prompt 结构 | 四层稳定性编排 | 明确优先级并利于稳定前缀 |
| 知识库权限 | 检索前授权 Collection 裁剪 | 防止越权数据进入 Prompt |
| MCP 身份 | 短期 HMAC JWT | 跨进程传递不可篡改身份 |
| 订单数据权限 | SQL 绑定 token.sub | 不信任模型生成的 userId |
| 安全评测 | 确定性规则 | 权限问题有明确真值 |
| 最终回答评测 | 暂未覆盖 | 当前接口返回决策和 MCP 结果 |

## 14. 当前局限

- 长期记忆尚未增加敏感信息分类器和人工编辑页面。
- 长期记忆第一版没有向量召回。
- 知识库权限只有 `GLOBAL/PERSONAL`，尚未扩展部门和租户 ACL。
- 内部 JWT 采用共享 HMAC，持有密钥的服务具备签发能力，尚未升级为非对称签名。
- 当前开发环境未接入 mTLS、工作负载身份、私网准入和 `jti` 重放检测。
- 当前评测主要使用合成数据，不能完全代表生产问题分布。
- 当前评测不覆盖最终自然语言回答质量。
- 10000 请求会产生模型 API 成本，需要单独运行并保存正式报告。

主动说明局限不会减分，反而能体现对工程边界和结果可信度的认识。

## 16. 面试追问速答

### 为什么要做长期记忆，直接保留聊天记录不行吗？

完整历史会无限增长，而且大量内容与当前问题无关。长期记忆只保存可复用的偏好、项目、约束和事实，再按相关性召回少量 Top N，能够控制 Token 和噪声。

### 为什么不把长期记忆放进 System Prompt？

长期记忆来自用户数据，可信度低于系统规则。放在 Semi-stable 用户上下文可以保持优先级边界，避免用户记忆污染系统指令。

### KV Cache 是你自己实现的吗？

不是。KV Cache 通常由模型服务提供。我做的是稳定前缀编排，将稳定规则放前面、动态证据放后面，提高支持 Prefix/KV Cache 的推理服务复用前缀计算的机会。

### 为什么 MCP 还要 JWT，直接传 userId 不行吗？

工具参数由 LLM 生成，不可信，普通 `userId` 也没有签名。JWT 将服务端确认的 `sub`、角色、Audience 和过期时间一起签名，模型无法篡改，远端服务也不需要共享 Ragent Session 就能独立验证。JWT 只传递可信身份，最终权限仍由工具授权和 SQL 行级过滤决定。

### 为什么不直接把用户登录 Token 转发给 MCP？

登录 Token 的受众是 Ragent，直接下发会扩大原始凭证的暴露范围。更合理的方式是由 Ragent 或企业 STS 将登录身份交换为面向目标 MCP 的短期 Token，只包含该服务需要的 Audience、Scope 和用户身份。

### JWT 能防止其他系统接入 MCP 吗？

不能单独做到。JWT 可以拒绝没有合法令牌的请求，但如果未知系统获得了有效 Token 或 HMAC 密钥，仍然可能调用服务。企业需要同时使用私网部署、API Gateway、mTLS 或工作负载身份、客户端注册和网络策略控制调用方准入。

### 企业里通常怎么做 Agent 到 MCP 的身份传递？

用户先通过 OIDC/SSO 登录 Agent，Agent Gateway 校验用户身份和工具权限，再通过 OAuth2 Token Exchange 或 On-Behalf-Of 获得目标 MCP 专用的短期 Token；服务间使用 mTLS，MCP 校验 Token 的签名、Audience、Scope 和用户 Claim，最后在业务及数据库层执行租户或用户隔离。

### 如果攻击者知道 MCP 地址会怎样？

地址本身不是凭证。当前 Order MCP 的 `/mcp` 没有 Bearer Token 会返回 401，伪造、过期、错误 Issuer 或错误 Audience 的 Token 也会被拒绝。生产环境还应让 MCP 只存在于私网，并通过安全组、NetworkPolicy、Gateway 和 mTLS 限制请求来源，避免把内部 MCP 直接暴露到公网。

### 为什么要双层鉴权？

Ragent 入口检查减少无权限调用，MCP 服务端鉴权才是最终安全边界。只做入口鉴权，绕过主应用后远端服务就失去保护。

### 如何证明普通用户没有越权？

评测订单包含确定性用户归属标记。评测器用不同用户身份从 Chat 入口发送本人、他人和管理员查询，只扫描 MCP `<data>` 返回体；任何其他用户订单、商品标记或用户 ID 都会触发零容忍安全失败。

### 你是如何设计这套评测的？

我先定义“普通用户只能看自己的订单”等业务不变量，再把它们拆成正常查询、他人订单、管理员诱导和并发稳定性场景。每条用例标注期望意图、是否调用 MCP、允许和禁止出现的订单归属，然后从 `/rag/eval` 进入完整 Chat 决策链路，分别计算意图、工具、语义、传输和越权指标，安全泄露采用零容忍硬门禁。

### 为什么安全判定不用 LLM Judge？

订单归属、用户 ID、订单状态和 `found` 都有确定性真值。规则判定成本更低、结果可复现，也不会因为 Judge 幻觉产生安全误报。LLM Judge 更适合后续评价最终自然语言回答的完整性和表达质量。

### 为什么要先跑 Smoke，再跑 1000 或 10000 请求？

大规模请求会放大配置错误并消耗模型额度。先通过单元测试、手工集成和 20 请求 Smoke 验证工具注册、身份透传与判定器，再运行标准和压力评测，才能让大规模结果具有解释价值。

### 为什么评测不直接调用 MCP？

直接调用 MCP 无法验证自然语言理解、问题改写、意图识别、工具选择和参数提取。从 Chat 决策入口进入才能覆盖 Agent 的完整执行决策。

### 这是不是完整端到端评测？

它覆盖自然语言输入到 MCP 数据结果，但不包含最终自然语言回答生成。后续需要增加同步回答接口或收集 SSE 回答，再评测事实一致性、完整性和表达质量。

### 首轮评测器为什么会误报？

一个原因是数字节点 ID 和 `intent_code` 格式不一致，另一个原因是把 `<question>` 中的订单号当成工具返回数据。修复后统一意图标签并只检查 `<data>`，同时增加了评测器自身的回归测试。
